Optimizing for big data

Clustered Indexes, Temporary Tables, and Good Design

From Stack Overflow user csano: "You have two tables, Customer and Order: Customer ---------- ID Name Address Order ---------- ID CustomerID Price If you wish to quickly retrieve all orders of one particular customer, you may wish to create a clustered index on the "CustomerID" column of the Order table. This way the records with the same CustomerID will be physically stored close to each other on disk (clustered) which speeds up their retrieval.""

With a huge database of millions of datapoints, queries can start taking a long time. Especially if you are working with multiple tables. A clustered index is a way to speed this up, as described above.

If you are interested in a particular set of datapoints in a database, and do not need to iterate through every point, you can create a temporary table that only includes the data you are working with. For example, say you have a number of linked indexes about your customers. You have millions of customers! (Yay!) You know things about them including their age, locations, language preference, as well as a TON of information about their buying history. Today you want to run a number of queries to see what you can learn about your customers over 65, in North America, and what kind of patterns you can find in their buying history. Rather than have each query iterate through the entirity of your tables, create a temporary table that only contains the information you plan on working with. It will take extra time up front, but could pay off if you are doing a number of queries.

Most importantly, if you are creating a database, make sure you are designing it well. Even if you don't expect your database to get huge, treat it as though it will. Limits on types of data to go in fields, connecting tables in the most effecient way possible, these things will pay off hugely down the road when you are sifting through your data.
